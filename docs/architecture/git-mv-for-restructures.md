# `git mv` for restructures

> When moving or renaming files, use `git mv` in a commit that contains
> **only** moves — no content changes. Pure-move commits preserve rename
> detection, which is what lets in-flight PRs rebase cleanly.

#pattern #git #workflow

## The rule

1. **Commit 1** — pure `git mv`. Nothing else. `git diff --stat` should
   show every entry as `file → new/path (100%)`.
2. **Commit 2+** — content changes (workspace Cargo.toml tweaks, path
   rewrites in scripts, doc updates). Separate from the move.
3. Announce in advance if there are open PRs touching affected paths;
   rebase (don't merge) once the restructure lands on main.

## Why

Git's rename detection uses content similarity. When the move commit has
the file *unchanged* at the new path, detection is 100% and any PR
that touches the old path rebases cleanly — git re-applies the PR's
patch at the new location automatically.

If the move commit *also* edits the file, similarity drops, rename
detection becomes flaky, and in-flight PRs land conflicts that must be
resolved by hand.

## Verified in practice

The monorepo restructure (commit `d0307ff` — moving `src/`, `assets/`,
and `Cargo.toml` into `desktop/`) was pure-move. A pending PR
(`#8 feat: refresh autogenerated task name word lists`, touching
`src/new_task_modal.rs`) was test-rebased onto the restructure branch
before push: git auto-detected the rename and applied the PR's 9-line
change at `desktop/src/new_task_modal.rs` with zero manual work.

## When moving larger chunks

- Split into multiple pure-move commits if staging matters (e.g.,
  "move crate A", then "move crate B").
- Use `git mv <dir> <new-dir>` — moves the whole tree in one go,
  still counted as renames by the detector.
- Avoid `git mv` on a file that the commit also modifies semantically
  (e.g., moving while changing the package name in `Cargo.toml`).
  Split into two commits.

## Anti-patterns

- **Delete + re-add.** Loses rename detection entirely. Every tool
  (history, blame, PR rebase) breaks.
- **Move + content edit in one commit.** Rename detection is
  best-effort; works sometimes, fails quietly others. Don't gamble.
- **Mixing many unrelated moves in one commit.** Makes review harder
  and increases the chance of a content-diff sneaking in that breaks
  a rename.

## When to help authors

After the restructure lands on main, help authors of still-open PRs
rebase. Usually `git rebase origin/main` is enough; if rename detection
doesn't fire automatically, `git rebase -Xfind-renames=50% origin/main`
helps. Worst case, point them at the specific file's new path.
